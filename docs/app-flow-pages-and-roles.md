# Page Purposes (One Line Each)

---

## 1. Landing

**Purpose:**
Introduce Strion as a structured visual performance system — not a self-improvement app.

Primary Goal:

* Convert to diagnostic start.

Core Elements:

* Hero with Strion Index preview
* System explanation (Diagnose → Execute → Optimize)
* Pricing
* Controlled CTA

---

## 2. Authentication

**Purpose:**
Secure account access with minimal friction.

Includes:

* Email + password
* Trial logic
* No distractions

Redirect logic:

* First-time → Onboarding
* Returning → Dashboard

---

## 3. Onboarding

**Purpose:**
Collect structural inputs and generate initial Strion Index.

Flow:

1. Photo Upload
2. Questionnaire
3. Diagnostic Result
4. Foundation Phase start

Emotional tone:

* Precision
* Clinical clarity
* No celebration

---

## 4. Dashboard (Gravitational Center)

**Purpose:**
Display real-time performance state.

Above the fold:

* Strion Index (dominant)
* Trend delta
* Sparkline

Below:

* Phase indicator
* Today’s missions
* Checkpoint countdown

This is the daily-use screen.

---

## 5. Protocol

**Purpose:**
Display personalized 90-day structure.

Sections:

* Phase 1 — Foundation
* Phase 2 — Structure
* Phase 3 — Refinement

Each week collapsible.
Tasks view-only.

---

## 6. Progress

**Purpose:**
Visualize longitudinal performance change.

Includes:

* Timeline (Day 0 → 14 → 30 → 60 → 90)
* Side-by-side comparison tool
* Index evolution chart

Data-forward.
No decorative effects.

---

## 7. AI Assistant

**Purpose:**
Provide structured analysis and recalibration insights.

Modes:

1. Protocol Mode
2. Recalibration Mode
3. Visual Review Mode

AI is analytical.
Never motivational.

---

## 8. Profile

**Purpose:**
Control user settings and system data.

Includes:

* Subscription management
* Data & photo privacy controls
* Account deletion
* Potential range visibility
* Logout

No vanity statistics.

# User Roles & Access Levels

---

## 1. Standard User (Primary Role)

Access:

* Onboarding
* Dashboard
* Protocol
* Progress
* AI Assistant
* Profile

Restrictions:

* Cannot alter Strion Index formula
* Cannot manually adjust protocol tasks
* Cannot edit recalibration logic

User executes.
System calibrates.

---

## 2. Admin (Internal Only)

Access:

* View anonymized structural metrics
* Monitor AI diagnostic outputs
* Adjust weighting model (server-side only)
* Manage subscription tiers

Admin tools never visible to user.

---

# Primary User Journeys (Max 3 Steps Each)

---

## Journey 1 — First-Time Optimization

1. Sign up
2. Complete onboarding diagnostic
3. Enter Dashboard (Phase 1 — Foundation)

Outcome:
User sees structured baseline and 90-day trajectory.

---

## Journey 2 — Daily Execution

1. Open Dashboard
2. Complete Today’s Missions
3. See subtle trend update

Outcome:
Controlled micro-progress without gamification.

---

## Journey 3 — Checkpoint Recalibration

1. Checkpoint notification
2. Upload updated photos
3. View recalibrated Index + protocol update

Outcome:
Measured structural correction.

---

## Journey 4 — Regression Handling

1. Dashboard shows negative projected trend
2. Open AI Assistant
3. Review execution variance explanation

Outcome:
User identifies friction, not failure.

---

## Journey 5 — Subscription Continuity

1. Trial nearing end
2. View pricing screen
3. Confirm subscription

No urgency timers.
No countdown theatrics.

---

# Navigation Structure (Mobile)

Bottom Nav:

* Dashboard
* Protocol
* Progress
* AI
* Profile

Rules:

* Dashboard always default
* No nested multi-layer menus
* Max 1 modal at a time
* No deep drill-down trees

Three taps max to any major function.

---

# System Logic Guardrails

* Index updates only at checkpoints
* Trend shifts reflect execution variance
* AI cannot override structural scoring
* Protocol changes only during recalibration

Strion diagnoses.
It does not punish.
